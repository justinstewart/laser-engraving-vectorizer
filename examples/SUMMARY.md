# Engraving Demo — Experiment Summary

Proof of concept for an AI-assisted logo-to-engrave-mask pipeline, driven conversationally through Claude Code.

## What We Built

A human-in-the-loop pipeline that takes a raster logo and produces a laser-ready SVG. The operator makes decisions at each step via labeled contact sheets generated by the pipeline; Claude Code orchestrates the scripts and presents the options.

## Final Outputs

Both final SVGs are in this directory:

- `starbucks-final.svg` — filled-mask engraving
- `firefox-final.svg` — line-art engraving

## Laser-Ready Criteria

From `LASER-SPECS.md`:

| Criterion | Target | Fail |
|-----------|--------|------|
| Nodes per path | < 200 | > 500 |
| Path type | Spline/Bezier curves | Polylines |
| Features | Single color, closed paths, no double lines | — |

The 500-node ceiling comes from GRBL controller buffer limits (2KB serial buffer). Exceeding it causes mid-path stalls and burn marks.

## Key Findings

### Different logos need different approaches

- **Two-color logos (Starbucks)**: Saturation or luminance thresholding produces a clean binary mask. Straightforward vectorization with vtracer.
- **Gradient logos (Firefox)**: Binary thresholding fails — all regions are equally saturated, and luminance can't separate the fox from the globe. Edge detection produces a line-art representation that preserves the logo's structure.

### Input downscaling beats post-processing simplification

We tried post-processing SVG paths through Douglas-Peucker + Bezier refit. The refit produced control points that overshot the original curve, causing visual artifacts (crossed lines, phantom shapes). Abandoned.

The working approach: downscale the binary mask before passing to vtracer. Fewer input pixels → naturally fewer nodes in the output. The operator picks the scale percentage that hits the node budget while preserving acceptable detail.

### Edge detection: PIDI wins

fal.ai's PIDI preprocessor (`fal-ai/image-preprocessors/pidi`) produces significantly cleaner, more connected edges than OpenCV adaptive thresholding or Canny. PIDI output is soft/grayscale — must threshold to binary before vectorizing. Threshold level matters: 120 preserves fine interior lines that 180 loses.

### Gradient logos are a different product

There's no single "right answer" for a gradient logo. The pipeline can produce:
1. **Line art engraving** — edge detection + vectorize (what we built for Firefox)
2. **Multi-layer depth engraving** — segment into regions, assign each to a different engrave power (attempted via K-means; didn't segment shapes cleanly)
3. **Dithered raster engraving** — grayscale + Jarvis dithering in LightBurn, not a vector workflow at all

The operator has to pick the engraving style before the pipeline can target it.

## The Bigger Lesson

The most useful finding wasn't about the vector pipeline — it was that **vector probably isn't the right output format at all**. Once you have a clean binary mask (step 3), sending it to LightBurn as a 1-bit raster sidesteps every problem that steps 4–5 try to solve: no node-count ceiling, no GRBL stalls, no vectorizer overshoot, no scale tuning. Laser controllers stream scanlines natively.

Vector earns its keep when you genuinely need vector output (cut paths, scoring lines, per-region power layers). For pure engraving from a clean mask, raster wins. The high-leverage problem is producing the clean mask — which is exactly where the AI tools (BiRefNet, PIDI) moved the needle.

## What We Didn't Solve

- **Multi-layer depth engraving** for gradient logos. K-means color quantization clustered by color similarity, not by shape, which fragmented the fox and globe incorrectly. Would require semantic segmentation (SAM, region-based models) to do well.
- **Post-processing SVG simplification.** The Bezier refit is broken in `lib/simplify.py`. Don't use without fixing.
- **Automated pass/fail validation.** We calculate node counts manually; a pipeline step that flags paths over 500 nodes would be straightforward to add.
